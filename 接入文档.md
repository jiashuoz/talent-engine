# talent-engine 接入文档

候选人简历与岗位匹配 API。上传简历 PDF 与岗位说明，返回结构化数据及匹配评分（含强项、差距、推荐意见）。

## 接入流程

1. **联系我们获取 API Key**（详见文末"联系方式"）。每个合作方分配独立 Key，便于用量追踪。
2. **测试 `/health` 确认网络连通**：
   ```bash
   curl https://<api-domain>/health
   # 期望返回：{"status":"ok"}
   ```
3. **将 Key 配置到您的服务端或小程序**。所有受保护接口需通过 `X-API-Key` 请求头携带。
4. **可选：用 Swagger UI 在线试调**。访问 `https://<api-domain>/docs`，点右上角 🔒 **Authorize** 粘贴 Key，即可在浏览器里直接调用任一接口。

> ⚠️ **请勿在客户端代码中明文存放 API Key**。微信小程序请将 Key 放在您自己的服务端转发，避免被反编译泄露。

## 鉴权

所有 `/v1/*` 接口需在请求头携带：

```
X-API-Key: mnk_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

- Key 缺失或无效 → `401 Unauthorized`
- 通过则放行，并将本次调用记录入用量统计。

## 接口清单

| 方法 | 路径 | 用途 | 是否阻塞 |
|---|---|---|---|
| `GET` | `/health` | 健康检查 | 同步 |
| `POST` | `/v1/resume/parse` | 解析简历 PDF | 同步 |
| `POST` | `/v1/job/parse` | 解析岗位说明文本 | 同步 |
| `POST` | `/v1/resume-matching/match` | 简历 × 岗位匹配评分 | 同步 |
| `POST` | `/v1/resume-matching/match/async` | 匹配评分（异步） | 异步 |
| `GET` | `/v1/resume-matching/match/{job_id}` | 查询异步结果 | 同步 |

完整交互式文档：`https://<api-domain>/docs`（Swagger UI）。机器可读 schema：`https://<api-domain>/openapi.json`。

## 详细接口说明

### 1. 解析简历 — `POST /v1/resume/parse`

**请求**（multipart/form-data）：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `resumes` | 文件（PDF）数组 | 是 | 单份 ≤ **5 MB**，单次最多 **20 份** |
| `resume_ids` | 字符串数组 | 否 | 客户端自定义 ID，缺省则使用文件名。若提供，长度需等于 `resumes`，且各值唯一 |

**示例**：

```bash
curl -X POST https://<api-domain>/v1/resume/parse \
  -H "X-API-Key: mnk_xxx..." \
  -F "resumes=@/path/to/陈先生.pdf"
```

**响应**（200）：

```json
{
  "status": "completed",
  "parsed": [
    {
      "resume_id": "陈先生.pdf",
      "filename": "陈先生.pdf",
      "resume": {
        "name": "陈先生",
        "gender": "男",
        "birth_year": null,
        "age": 28,
        "phone": "138xxxx0000",
        "email": "chen@example.com",
        "hometown": null,
        "education": [
          { "school": "...", "degree": "本科", "major": "...", "start": "2014.09", "end": "2018.06", "gpa_or_rank": null }
        ],
        "experience": [
          { "organization": "...", "title": "...", "start": "2018.07", "end": "至今", "description": "..." }
        ],
        "certifications": ["..."],
        "skills": ["..."],
        "languages": [],
        "self_evaluation": "...",
        "raw_text": "完整简历原文..."
      }
    }
  ],
  "errors": [],
  "stats": {
    "files_received": 1,
    "parsed_ok": 1,
    "parsed_failed": 0,
    "elapsed_ms": 7152,
    "input_tokens": 700,
    "output_tokens": 517
  }
}
```

字段说明：

- 所有结构化字段（除 `name` 外）都可能为 `null` 或空数组——简历未提供即缺省，不报错。
- `raw_text` 是 PDF 文本提取的原始内容，包含简历全文。**含 PII，请按贵司隐私要求处理后再展示或存储**。
- 单份简历解析失败（PDF 损坏、图片型 PDF 无法提取文字、内容过短等）会出现在 `errors[]` 中，HTTP 状态仍为 200。整体请求级错误（鉴权失败、文件过大、超过数量上限）才返回 4xx。

### 2. 解析岗位说明 — `POST /v1/job/parse`

**请求**（multipart/form-data）：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `jds` | 文件（.txt）数组 | 是 | 单份 ≤ **200 KB**，单次最多 **10 份**。每份可包含一个 JD 或多个用 `招聘单位` 分隔的 JD |

**示例**：

```bash
curl -X POST https://<api-domain>/v1/job/parse \
  -H "X-API-Key: mnk_xxx..." \
  -F "jds=@/path/to/岗位说明-运维.txt"
```

**响应**（200）：

```json
{
  "status": "completed",
  "parsed": [
    {
      "job_id": "岗位说明-运维.txt#0",
      "filename": "岗位说明-运维.txt",
      "chunk_index": 0,
      "job": {
        "company": "神思电子",
        "position": "网络运维工程师",
        "location": "济南市高新区",
        "education_min": "专科",
        "age_min": null,
        "age_max": 45,
        "majors_preferred": ["计算机相关专业"],
        "experience_years_min": 3,
        "gender_preference": null,
        "height_min_cm": null,
        "certifications_required": [],
        "image_requirements": null,
        "duties": ["从需求到运维的全流程产品设计..."],
        "salary_min": 5000,
        "salary_max": 6000,
        "work_schedule": "周内调休，上12休36，每月需工作满180小时",
        "benefits": ["五险一金", "节假日福利"],
        "raw_text": "【招聘单位】：..."
      }
    }
  ],
  "errors": [],
  "stats": {
    "files_received": 1,
    "chunks_detected": 1,
    "jobs_parsed": 1,
    "jobs_failed": 0,
    "elapsed_ms": 4978,
    "input_tokens": 828,
    "output_tokens": 367
  }
}
```

字段说明：

- **`company` 与 `position` 是必填**，其余字段缺失时为 `null` 或空数组。
- `job_id` 格式为 `<原文件名>#<chunk_index>`——若一份 .txt 内含多个 JD（用 `招聘单位` 分隔），`parsed` 会返回多条，`chunk_index` 自 0 起递增。
- `raw_text` 是该 chunk 在原文件中的对应片段，便于回查原文。

### 3. 简历 × 岗位匹配 — `POST /v1/resume-matching/match`

将上述两个解析接口返回的 `resume` / `job` 对象直接喂入即可，无需二次转换。

**请求**（application/json）：

```json
{
  "resumes": [
    { "resume_id": "陈先生.pdf", "resume": { ... } }
  ],
  "jobs": [
    { "job_id": "岗位说明-运维.txt#0", "job": { ... } }
  ],
  "options": {
    "concurrency": 4
  }
}
```

`options.concurrency` 为可选项：控制后端对 LLM 的并行调用数。缺省由服务端选择合理值；显式设置仅在调优时需要，多数情况下可省略。

**响应**（200）：

```json
{
  "status": "completed",
  "matches": [
    {
      "resume_id": "陈先生.pdf",
      "job_id": "岗位说明-运维.txt#0",
      "score": 30,
      "verdict": "不推荐",
      "hard_fails": ["最高学历为中专，未达到岗位要求的专科及以上学历"],
      "strengths": ["有餐厅服务员经验，具备基本的顾客接待和沟通能力"],
      "gaps": [
        "缺少专科及以上学历证书",
        "无任何与运维相关的工作经验"
      ],
      "reasoning": "候选人最高学历为中专..."
    }
  ],
  "errors": [],
  "stats": {
    "pairs_scored": 1,
    "pairs_failed": 0,
    "elapsed_ms": 3430
  }
}
```

字段说明：

- `score`：**0–100 整数**，综合评分。
- `verdict`：中文短语，常见值有 `强烈推荐` / `推荐` / `可考虑` / `勉强` / `不推荐` 等。**该字段由 LLM 自由生成，不是固定枚举**——业务判断请优先依赖 `score` 与 `hard_fails`，`verdict` 仅作展示。
- `hard_fails`：硬性条件不符项（学历、年龄、必备资格证等）。非空通常意味着不应录用。
- `strengths` / `gaps`：相对软性的强项与待补足项，含改进建议。
- `reasoning`：综合说明，可直接展示给招聘者。
- `errors[]`：**单对评分失败也会出现在这里**（HTTP 仍 200）。每项含 `resume_id`、可选 `job_id` 与 `error` 描述。前端应同时渲染 `matches` 与 `errors`，对失败项做降级处理。

### 4. 异步匹配 — `POST /v1/resume-matching/match/async`

当需要批量匹配（例如 50 份简历 × 5 个岗位 = 250 对）时，同步接口可能超过小程序的 60 秒请求超时。请改用异步：

```bash
# 1. 提交任务
curl -X POST https://<api-domain>/v1/resume-matching/match/async \
  -H "X-API-Key: mnk_xxx..." \
  -H "Content-Type: application/json" \
  -d @match_payload.json
# 响应（HTTP 202）：
# {"job_id": "rmj_3m4ZaihILu...", "status": "queued"}

# 2. 轮询结果（建议 2 秒一次）
curl https://<api-domain>/v1/resume-matching/match/<job_id> \
  -H "X-API-Key: mnk_xxx..."
```

**状态机**：

| `status` | 含义 | 还会带哪些字段 |
|---|---|---|
| `queued` | 任务已入队，尚未开始 | — |
| `running` | 后端正在评分 | `progress: { pairs_done, pairs_total }` |
| `completed` | 全部评分结束 | `matches`、`errors`、`stats` |
| `failed` | 任务级失败（罕见） | `error`（失败原因字符串） |

> 轮询时若得到 `404 Job not found or expired`，表示 `job_id` 不存在或服务端结果已过期清理——请重新提交。

**完成态响应示例**：

```json
{
  "job_id": "rmj_3m4ZaihILu...",
  "status": "completed",
  "matches": [ /* 同步接口的 matches 数组 */ ],
  "errors": [],
  "stats": { "pairs_scored": 250, "pairs_failed": 0, "elapsed_ms": 187320 }
}
```

## 调用上限

| 项 | 上限 |
|---|---|
| 单份简历 PDF | **5 MB** |
| 单次 `/v1/resume/parse` 文件数 | **20 份** |
| 单份 JD 文本 | **200 KB** |
| 单次 `/v1/job/parse` 文件数 | **10 份** |
| 单次 `/v1/resume-matching/match` 简历数 | **100** |
| 单次 `/v1/resume-matching/match` 岗位数 | **100** |
| 单次 `/v1/resume-matching/match` 总对数（简历 × 岗位） | **1000** |
| 推荐请求并发 | 同步 ≤ 5；异步任务数 ≤ 3（指引值，非硬限） |

超出上限会立即返回 `413 Payload Too Large`，含具体说明（例如 `max 20 resumes per request`）。

## 性能与冷启动

正常情况下：

- `/v1/resume/parse`（每份简历）：约 **3–8 秒**
- `/v1/job/parse`（每份 JD 文件）：约 **3–6 秒**
- `/v1/resume-matching/match`（每对简历×岗位）：约 **3–10 秒**，并行评分
- `/v1/resume-matching/match/async` 提交：< 100 ms（任务入队即返回 `job_id`）

**冷启动延迟（首次调用慢）**：服务底层数据库采用按需计费、空闲自动暂停模式。如果距上次任意调用超过约 10 分钟，**首次请求会额外耗时 10–20 秒**（数据库需唤醒）；之后请求恢复正常。

建议：
- 高频场景（小程序前台直接调用）：业务高峰前可先调用一次 `/health` 预热，规避用户感知到的冷启动。
- 低频场景（管理后台批量任务）：把首次调用的超时设为 ≥ 30 秒，并对超时做一次重试。

## 微信小程序集成

### 路径 A（推荐，仅限部署在 微信云托管）：`wx.cloud.callContainer`

若 talent-engine 部署在 **微信云托管**，小程序可直接通过 `wx.cloud.callContainer` 调用，**无需配置 `服务器域名` 白名单、无需重新提交审核**——这是 微信云托管 相对于其它部署方式的最大优势。

```js
wx.cloud.callContainer({
  config: { env: '<your-env-id>' },
  path: '/v1/resume-matching/match',
  method: 'POST',
  header: { 'X-API-Key': 'mnk_xxx...' },  // 仍建议由服务端代理
  data: { resumes, jobs },
  success(res) { console.log(res.data.matches); }
});
```

### 路径 B（其它部署方式）：`wx.request` + 域名白名单

若 API 部署在其它云厂商（UCloud / Aliyun / Tencent CVM 等），则需走标准的小程序网络请求流程：

1. 在小程序 **管理后台 → 开发管理 → 开发设置 → 服务器域名** 中将 `https://<api-domain>` 添加到 `request 合法域名` 列表。
2. 提交小程序新版本审核（每次新增域名都需要）。
3. 通过 `wx.request` 调用。

```js
wx.request({
  url: 'https://<your-server>/api/match',  // 推荐：自家服务端转发
  method: 'POST',
  data: { resumes, jobs },
  success(res) { console.log(res.data.matches); }
});
```

⚠️ **API Key 不应放在小程序前端代码里**。无论走哪条路径，都建议由您自己的服务端持有 Key 并代为转发，避免被反编译泄露。

## 错误码

| HTTP | 含义 | 处理建议 |
|---|---|---|
| 200 | 请求成功（部分子任务可能失败，见 `errors[]`） | 同时渲染 `matches/parsed` 与 `errors` |
| 400 | 参数错误（空数组、ID 重复、长度不匹配等） | 检查请求体 |
| 401 | API Key 缺失或无效 | 检查请求头 |
| 404 | 异步 `job_id` 不存在或已过期 | 重新提交任务 |
| 413 | 文件过大、文件数量超限或匹配对数过多 | 拆分请求 |
| 422 | 请求体格式不符 schema（FastAPI 校验） | 对照 `/docs` 字段定义 |
| 500 | 服务端异常（含 LLM 失败） | 重试一次；持续失败请联系我们 |

## 常见问题

**Q：返回的 `score` 是怎么算的？**
A：综合学历、工作经验、技能匹配度、稳定性、年龄等多维度由 LLM 综合给出。硬性条件不符（如学历低于岗位最低要求）会显著拉低分数；软性差距由 LLM 综合判断。具体维度可能随模型与 Prompt 迭代调整。

**Q：可以批量上传多份简历吗？**
A：可以。`resumes` 字段为数组，单次最多 20 份（解析）/ 100 份（匹配）。返回的 `parsed` / `matches` 数组顺序对应输入。

**Q：解析的中间结果（结构化简历）能存下来重复使用吗？**
A：**可以，且强烈推荐这么做**。`/v1/resume/parse` 返回的 `resume` 对象、以及 `/v1/job/parse` 返回的 `job` 对象，可直接喂回 `/v1/resume-matching/match`，无需每次重新解析 PDF。这能显著降低延迟与成本。

**Q：同一份 PDF 重复 `parse` 会怎样？**
A：服务端不做幂等去重——每次调用都会重新走 LLM，按用量计费。请在贵司侧根据 `resume_id` 或文件哈希做缓存。

**Q：扫描件 / 纯图片型 PDF 能解析吗？**
A：**目前不支持**。我方使用 `pypdf` 提取文本，对图片型 PDF 会失败并出现在 `errors[]` 中。如有需求请联系我们（可上游接入 OCR）。

**Q：数据保留多久？**
A：API 用量统计（含调用方、token 数、耗时）默认保留 12 个月。**原始简历内容（PDF、`raw_text`）不持久化**，仅在请求生命周期内驻留。异步任务结果在内存中保留有限时间，超时后查询返回 404。

**Q：单次请求超时上限？**
A：同步接口建议在 60 秒内完成（小程序网络请求默认上限）。批量过大时请改用 `/match/async`。
